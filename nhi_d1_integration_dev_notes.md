# NHI 健保用藥資料自動化與 Cloudflare D1 整合 - 完整開發筆記

## 專案背景與目標
專案的原始起點是因為需要定期獲取衛福部中央健康保險署的「健保用藥品項」資料。從最初單純寫 Python 腳本手動下載與清洗，隨後封裝成可攜式的桌面執行檔與 AI 專屬的自動化 Skill，最終演進到使用 GitHub Actions 達成全自動無人值守，並將清洗後的資料寫入 Cloudflare D1 資料庫，供前端產品呼叫。

本篇筆記將詳細記錄這一路以來的四個核心開發階段。

---

## 第一階段：資料獲取與基礎清洗 (Python 腳本)
*   **目標**：撰寫腳本從政府開放資料平台下載最新的健保用藥品項，並初步清理。
*   **實作細節**：
    *   **下載機制**：撰寫 `download.py` 等腳本串接政府公開資料 API (或網頁)，抓取最新的 ZIP 壓縮檔並解壓縮出 CSV。
    *   **核心清理邏輯**：撰寫 `clean_data.py` 和後續整合的 `process_nhi.py`，專注於解決資料格式不一致的問題（如去除 BOM、處理引號）。
    *   **過濾無效品項**：開發階段發現有些品項的支付價格為 0，這對前端應用不具參考價值。因此特別加入邏輯（及 `--exclude-zero` 參數）將這些品項予以剔除，讓最終輸出的 `cleaned_nhi_data_no_zero.csv` 筆數從約 1.6 萬降至 1.3 萬多筆。

## 第二階段：桌面工具化 (建立免安裝執行檔)
*   **目標**：讓沒有安裝 Python 環境的使用者也能一鍵完成資料下載與清洗。
*   **實作細節**：
    *   **PyInstaller 封裝**：使用 PyInstaller 將整理好的 `process_nhi.py` 腳本打包。
    *   **Spec 檔微調**：建立 `process_nhi.spec` 設定檔以處理打包過程中遺漏的隱藏依賴或是靜態資源。
    *   **產出物**：最終在 `dist` 資料夾中產出了一個獨立的 `process_nhi.exe`（桌面執行檔），使用者雙擊或在終端機輸入指定參數即可運作。

## 第三階段：打造 AI 專屬自動化流程 (Gemini Skill)
*   **目標**：將這個處理流程教給 AI，讓使用者只要對話輸入一句「幫我抓最新健保藥品資料」，AI 就能幫忙代勞。
*   **實作細節**：
    *   **建立專屬 Skill**：在開發環境的 `skills` 目錄下建立 `nhi-drug-processor` 自訂技能。
    *   **撰寫 SKILL.md**：使用 Markdown 與 YAML 定義此技能的使用時機（例如：當使用者提到 "process NHI data" 時觸發）、所需的工具存取權，以及 AI 應該要執行的固定 SOP。
    *   **成果**：成功讓工作流（Workflow）腳本化，AI 可以自主幫忙呼叫打包好的工具或腳本來產出最新 CSV。

## 第四階段：無伺服器架構升級 (GitHub Actions + Cloudflare D1)
*   **目標**：徹底解放雙手！將流程搬上雲端定時觸發，並轉存為資料庫 API 供外部存取。
*   **專案結構**：新建 `nhi-cloud-action` 專案庫。

### 1. D1 寫入機制的演進
這是專案中卡關較久且重構幅度最大的部分：
*   **嘗試 A：REST API (失敗)**
    *   嘗試直接呼叫 Cloudflare D1 REST API 發送 Batch Insert SQL 指令。
    *   遭遇 HTTP 400 錯誤 (`Invalid input: Expected object, received array`)，受限於 D1 API 對大批次 Payload 格式的嚴格驗證。
*   **嘗試 B：使用 Wrangler CLI 搭配 SQL 檔案 (成功解決)**
    *   改變策略，先由 Python 將原本的 CSV 內容動態轉換成標準的 SQL `INSERT` 語句，並匯出成 `import.sql`。
    *   為了讓 CI 環境的 Wrangler 不需要全域設定檔就能成功執行，我們在腳本內**動態產生一份臨時的 `wrangler.toml`**。
    *   最後透過 `subprocess.run` 自動呼叫 `npx wrangler@latest d1 execute nhi-drugs --remote --file=import.sql` 完美將近萬筆資料穩固推上雲端！

### 2. GitHub Actions 排程與狀態追蹤
*   **排程時間考量**：我們在 `.github/workflows/daily_nhi_job.yml` 中將 Cron 設定為 `0 18 * * *` (UTC)，對應台灣時間為**每天凌晨 2:00**，避開前端系統可能的使用尖峰。
*   **同步日誌表 (`sync_logs`)**：為了讓前端明確顯示資料的「最後更新時間」，我們在匯入 SQL 的最後面加入了一段小巧思——建立並塞入一筆成功紀錄到 `sync_logs` 資料表中。
    *   包含欄位：`sync_time` (執行時間戳)、`status` ('Success')、`total_records` (處理筆數)。

## 踩坑紀錄與限制摘要 (雷區注意)

**⚠️ Cloudflare D1 免費版 (Free Tier) 用量限制**
*   **限制內容**：D1 免費方案**每日寫入列數上限為 100,000 筆** (100k rows/day)。
*   **踩坑實況**：由於 GitHub Actions 每次執行都會進行「全量更新」(寫入約 1.4 萬筆)。在除錯過程中，因連續手動觸發 Action 進行測試將近 10 次，當天的免費額度 (`138,960 / 100,000`) 便被消耗殆盡，導致 D1 鎖定寫入權限。
*   **解決與防範**：這只是日限額，每日台灣時間早上 8 點會自動重置。未來排程每日僅於凌晨執行一次 (耗用 1.4 萬額度)，日常營運完全足夠。未來測試類似的全量匯入腳本時，請先將測試用 CSV 截短至 10~50 筆內。

## 總結
本專案經歷了「**小工具腳本 -> 桌面執行檔 -> AI 整合技能 -> 全自動雲端排程**」的完整演進，展現了工作流逐步自動化、現代化的過程。未來前端團隊只需專注在應用層，透過 D1 API (或中介層) 提取資料並關聯 `sync_logs` 即可提供用戶最即時的健保用藥資訊。
